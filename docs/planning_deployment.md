## Целевой результат

### Как пользователь видит фронт (2026-06-26)

**Форма генерации (Отчётность), одна операция за дату:**

1. Указывает **дату** (за которую обрабатываются отчёты).
2. **Загружает всё, что в референсе выбиралось через «путь»**:
   - **отчёты `.docx`** — **мультивыбор**, как на `/daily`;
   - **Приложение 7** (`.xlsm`);
   - **шаблон расстановки** (`.xlsm`) — **обязателен** (ежедневный файл на отправку).
3. Нажимает **«Сформировать»**.
4. Скачивает **ZIP**:

| Этап | Файл на выходе |
|------|----------------|
| Парсинг | `summary_ДД.ММ.ГГГГ.xlsx` |
| Прил.7 | заполненный `.xlsm` |
| расстановка | **только `.xlsm`** |

**Планирование** — справочники; в форму генерации не входят (PostgreSQL).

**Лог** шагов в UI — как на `/daily` (прогресс, ошибки сверки).

Запуск — **только вручную из UI** (без cron / daily_agent).

## Что делает референс (логика, не стек)

| Шаг | Скрипт | Вход | Выход |
|-----|--------|------|-------|
| 1 | `collect_data.py` | папка `.docx` | `summary.xlsx`: Файл, Дата, Объект, Инженер СК, Генподрядчик |

| 3 | `generate_rasstanovka.py` | Прил.7 + справочники | HTML/отчёт расстановки |
| — | `fill_aos_personal.py` | лист «Отчёт о ВОУ» Прил.7 | отдельный лист **«АОС_Персонал»** (свод по разделам ИП/ТР/КП…) |

Референс: Flask, JSON (`справочник.json`, `должности.json`), на Windows — Excel COM для `.xlsm`.

### Пояснение: «АОС Персонал»

В референсе это **отдальный лист** в книге Excel: из уже заполненного Прил.7 (лист «Отчёт о ВОУ») собирается сводная таблица по разделам (ИП, ТР, КП, ОЭ, ПО) — кто сколько дней отработал по блокам. Основной поток (summary → Прил.7 → расстановка) **без этого листа работает**. Если понадобится — отложенная фаза после v1.

### Фаза 1 — справочники в БД

**1.1 Таблица должностей**

```text
positions
  id          — slug или uuid
  title       — «Инженер СК (общестроительные работы)»
  description — многострочный текст для расстановки / Прил.7
  sort_order  — порядок в отчёте (как _DOLZH_ORDER в референсе)
  is_active
```

- Seed из `Noviy proekt AI/src/должности.json` (разово).  
- UI: просмотр / редактирование в разделе **Планирование** (или подраздел «Должности»).  
- `personnel.position` → либо FK `position_id`, либо хранить title и валидировать по справочнику при импорте (решить при реализации).

**1.2 Подрядчики**

- Импорт из Excel пользователя → `contractors`.  
- Поля уточнить по файлу (минимум: name; возможно aliases для нормализации из docx, template_stem).  
- Логика `CONTRACTOR_MAP` из `collect_data.py` → конфиг в БД или таблица `contractor_aliases`, не хардкод в Python.

**1.3 Проекты**

- Импорт из Excel пользователя → `projects` + при необходимости `project_engineers`.  
- Сопоставление «Объект» из docx с `projects.object_name` / `title` — правила строгой сверки (фаза 0).

**Выход:** API + UI импорта/редактирования; RelaxDev с актуальными данными.

### Фаза 2 — парсинг docx → summary

**Модуль:** `sk_reporter/planning/` (новый пакет).

- Парсинг таблиц docx — логика как `collect_data.py`: Дата, Объект, Инженер СК, Генподрядчик.  
- Нормализация подрядчика — по справочнику БД (не `CONTRACTOR_MAP` в коде).  
- **Строгая сверка:**  
  - ФИО → запись в `personnel` (иначе ошибка в отчёте UI, файл не включается или весь прогон стоп — решить).  
  - Объект → проект в БД (аналогично).  
  - Подрядчик → `contractors` (аналогично).  
- Экспорт `summary_*.xlsx` — openpyxl, без pandas на проде.

**UI:** страница загрузки (мульти-select `.docx` или ZIP), прогресс, лог ошибок сверки, кнопка «Скачать summary».

**Выход:** рабочий шаг 1 без Прил.7.

### Фаза 3 — заполнение Приложения 7

- Вход: **загруженный пользователем** `.xlsm` + summary с фазы 2.  
- openpyxl: **только значения ячеек**, `keep_vba=True` при сохранении, макросы не перезаписывать.  
- Логика как `fill_pril7.py`: лист «Отчет о ВОУ», строка дат = 12, колонки F–AJ, сумма AK; 1/N; вставка строк.  
- Описания действий инженера — из `positions.description` по должности сотрудника.

**Выход:** заполненный `.xlsm` в ZIP.
